About the project

Stride is a fitness-tracking platform. The backend is a Node.js + TypeScript service running on Express, backed by Postgres 15 (managed via Prisma), with Redis for sessions and BullMQ for background jobs. We deploy to AWS ECS via GitHub Actions.

The team is currently 9 engineers split across two pods: Platform (auth, observability, infra) and Product (workouts, social, feeds). The auth/social systems were largely written by Diego (now on parental leave); the workout module is owned by Priya. Direct questions about the workout schema to Priya in #stride-product.

A short history of why things are the way they are

We originally built on Sequelize but migrated to Prisma in Q3 2024 after the third major migration footgun (Sequelize would silently drop NOT NULL on column rename — ask Diego if you want the war story). The Prisma migration took ~6 weeks; some legacy code in src/legacy/ still uses raw SQL queries which we’re slowly porting.

We tried microservices in 2023 and rolled it back six months later (Conway’s Law won). The repo is intentionally a modular monolith now. Don’t propose splitting it without an RFC.

The session store moved from in-process LRU to Redis after a P0 in November 2024. There is still a TOKEN_CACHE export in src/middleware/auth.ts for backwards compatibility — it’s deprecated but not yet removed because three internal tools still import it.

Architecture overview

src/
├── api/        — HTTP handlers, one file per resource
├── services/   — business logic, called from handlers
├── db/         — Prisma client, transaction wrappers, raw escape hatch
├── jobs/       — BullMQ workers and schedules
├── middleware/ — auth, rate-limit, request-id, observability
├── lib/        — shared utilities (validators, formatters, error types)
├── types/      — shared TypeScript types
└── legacy/     — pre-Prisma code being slowly retired

Handlers should be thin (parse → call service → return). Services should be testable without HTTP. Database access goes through Prisma except in src/legacy/ and src/db/raw/.

How to develop

We use pnpm (npm and yarn will break our lockfile). Common commands:

pnpm install         # one-time setup
pnpm dev             # local dev server with hot reload
pnpm test            # vitest, watch mode in dev
pnpm test:ci         # vitest single run for CI
pnpm lint            # eslint + prettier check
pnpm lint:fix        # eslint + prettier write
pnpm typecheck       # tsc --noEmit
pnpm build           # esbuild output to dist/
pnpm gen             # regenerate Prisma client + OpenAPI types
pnpm migrate:dev     # apply Prisma migrations to local DB
pnpm migrate:prod    # apply Prisma migrations to prod (DO NOT RUN LOCALLY)

Local DB is in docker-compose.yml. Run docker compose up -d db redis first.

Coding standards

We try to follow these:

  • Write good code. Be careful with security — we had a token-leak incident in 2023 and are still recovering trust internally.
  • Handle errors. Use our custom error types in src/lib/errors.ts (AppError, NotFoundError, ValidationError).
  • Write tests for new features. Aim for 80% coverage on new code.
  • Use descriptive variable names — no single letters except in tight loops.
  • Use 2-space indentation, single quotes, semicolons. Prettier config is in .prettierrc.
  • Avoid any in TypeScript. Use unknown and narrow.
  • Follow the existing patterns in the codebase. If you’re adding something new, look at how a similar thing is done first.
  • Write JSDoc on exported functions for non-obvious behavior.
  • Prefer named exports over default exports.
  • Files under src/generated/ are autogenerated by pnpm gen. Never edit them.
  • Don’t use --no-verify on commits. The pre-commit hooks are load-bearing (see .husky/pre-commit).

Adding a new endpoint

  1. Create the handler in src/api/<resource>.ts.
  2. Register the route in src/app.ts.
  3. Add the request/response types in src/types/<resource>.ts.
  4. Add the service in src/services/<resource>.ts.
  5. Write a test in src/api/<resource>.test.ts.
  6. Update docs/openapi.yaml (or run pnpm gen to regenerate from types).
  7. Add the new endpoint to the rate-limit config in src/middleware/rate-limit.ts.
  8. If the endpoint requires auth, add it to the protected routes list in src/middleware/auth.ts.

Adding a database migration

  1. Edit prisma/schema.prisma with your changes.
  2. Run pnpm prisma migrate dev --name <descriptive_name> — this creates a migration file and applies it locally.
  3. Inspect the generated SQL in prisma/migrations/.
  4. Write a rollback in the Down section. Do not leave -- TODO.
  5. Test the rollback works locally: pnpm prisma migrate resolve --rolled-back <name>.
  6. Open a PR — migrations require security team review.
